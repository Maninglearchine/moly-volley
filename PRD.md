## 1. 문서 개요

### 배경 및 목적
피카츄 발리볼(1997)의 오픈소스 웹 재현 버전(pikachu-volleyball)을 기반으로, 캐릭터 이미지만 커스텀 에셋으로 교체한 새로운 웹 브라우저 게임을 제작한다. 게임 로직·물리 엔진·AI는 원본 소스를 그대로 활용하여 개발 범위를 이미지 교체 및 UI 커스터마이징으로 최소화한다.

### 성공 기준
- 원본 게임과 동일한 플레이 경험 유지
- 커스텀 캐릭터 이미지로 자연스럽게 교체
- 브라우저에서 빌드 없이 바로 실행 가능 (또는 `npm start`로 실행)

## 5. 스프라이트 교체 상세 설계

### 5-1. 원본 스프라이트 시트 구조 분석

원본 `sprite_sheet.png`(476×885px)에 포함된 주요 프레임:

#### 플레이어 애니메이션 상태 (7가지)
| 상태 | 설명 | 프레임 수 |
|------|------|----------|
| 0 | 대기 (서 있음) | 1 |
| 1 | 달리기 | 2 |
| 2 | 점프 중 | 1 |
| 3 | 공 히트 | 2 |
| 4 | 파워 히트 | 1 |
| 5 | 다이빙 | 1 |
| 6 | 넘어짐 | 1 |

#### 공 애니메이션 상태 (6가지)
| 상태 | 설명 |
|------|------|
| 0~4 | 회전 프레임 (rotation 값 직접 사용) |
| 5 | 하이퍼 볼 (파워 히트 후) |

### 5-2. 커스텀 스프라이트 제작 가이드

#### 접근 방식: 방식 A (스프라이트 시트 직접 교체)
원본 sprite_sheet.json의 프레임 좌표를 유지하면서 해당 위치에 새 이미지를 그려 넣는 방식.

**단계:**
1. 원본 `sprite_sheet.json` 파싱하여 각 프레임 좌표·크기 확인
2. 빈 캔버스(476×885) 생성
3. img1.png (우측 플레이어), img2.png (좌측 플레이어) 리사이즈 → 각 상태별 프레임으로 배치
4. img0.png (공) 리사이즈 → 공 프레임 6개 위치에 배치 (회전 효과는 canvas transform 적용)
5. 배경 요소(지면, 그물, 구름 등)는 원본에서 그대로 복사

#### 접근 방식: 방식 B (view.js 수정으로 이미지 직접 로드) ← 권장
`view.js`의 `makePlayerAnimatedSprites()` 및 `makeBallAnimatedSprites()`를 수정하여 스프라이트 시트 대신 개별 이미지 파일을 로드.

**장점**: 스프라이트 시트 편집 불필요, 이미지 교체만으로 구현  
**단점**: 애니메이션 프레임이 1장(정지 이미지)으로 단순화됨

**권장 구현 방식**: 방식 B로 빠르게 구현 후, 필요시 방식 A로 고도화

### 5-3. 방식 B 구현 계획

```
src/resources/assets/images/
├── sprite_sheet.png        (원본 유지 - 배경·UI 요소용)
├── sprite_sheet.json       (원본 유지)
├── custom/
│   ├── player_left.png     ← img2.png (좌측 플레이어)
│   ├── player_right.png    ← img1.png (우측 플레이어)
│   └── ball.png            ← img0.png (공)
```

**view.js 수정 포인트:**
- `makePlayerAnimatedSprites()`: 커스텀 이미지 텍스처로 Sprite 생성
- `makeBallAnimatedSprites()`: 커스텀 볼 이미지 텍스처로 Sprite 생성
- 이미지 스케일 조정: 캐릭터 64×64px, 볼 40×40px(반지름 20 기준)

## 8. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 이미지 배경 제거 필요 | 중 | Canvas API로 배경 마스킹 또는 PNG 투명도 활용 |
